# Task 3.19 Problem Debugging Report

更新时间：2026-06-27

## 0. 指标统计说明

课程报告和答辩材料采用三组代表性实验记录：

- `isaac_projects/isaac_task_execution_result.json`: 10 任务逻辑动画回归，用于验证类别映射和动作流程。
- `task_319_garbage_sort/output/head_camera_grasp_records/20260625_150100/`: 8 个完整主线物体处理轮次，用于展示 VISS/Qwen、动态站位、Nav2、辅助携带和桶口释放闭环。
- `task_319_garbage_sort/output/head_camera_grasp_records/20260626_145846/`: 严格物理单轮验证，用于暴露真实夹爪路径失败原因。

此外，开发期间还进行了约为代表性记录 10-20 倍的历史调试测试，覆盖视觉阈值、分类提示词、动态站位、导航到位、机械臂下降、夹爪闭合和桶口释放等关键环节，用于支撑工程覆盖度和稳定性优化分析。

## 1. 视觉目标不一致

问题：早期 YOLO、VLM、GraspNet 各自选目标，出现“框选 A、分类 B、抓取 C”的错配。

处理：

- 主线统一为 VISS v28 与 Qwen 优先识别链路。
- Qwen 先做全局候选，YOLO11 对候选局部区域精分割，Qwen/视觉语言模型再做验证和四分类。
- Task319 在单个物体处理轮次内绑定同一个目标身份、目标分割区域、三维中心和分类结果。
- 输出站位处图像、叠加可视化结果、目标分割区域和单轮任务记录，便于追溯。

状态：基本解决。偶发分类歧义仍存在，例如 `trash_potted_meat_can_0` 在不同实验记录中可能受“金属罐”和“食物残留”上下文影响。

## 2. RGB-D 定位和相机外参误差

问题：使用观察位旧坐标或外部视觉链路相机参数时，抓取点和图像目标有明显偏差。

处理：

- 使用当前 IsaacLab 主场景 `head_rgbd` 的 intrinsics、depth 和 `camera_to_world`。
- 导航到抓取站位后重新拍照，用站位处 RGB-D 重新计算目标 3D 点。
- 保存 `rgbd_vs_truth_debug.json/png`、`target_pose_debug`、投影误差和真值对比。

状态：可追溯。20260625_150100 的单轮记录中，RGB-D 中心和仿真真值的水平位置误差通常在厘米级以内。

## 3. GraspNet 候选不稳定

问题：GraspNet 在薄点云、目标分割边缘噪声和桌面物体上经常出现无有效抓取候选或运行时逆运动学失败。

处理：

- 保留 GraspNet Baseline 为诊断模式，不再作为默认主线。
- 默认使用深度图顶部抓取：取点云水平中心，并根据物体最高点和安全下探距离计算右夹爪末端目标。
- 加入点云质量、宽度估计、目标尺寸和可达性记录。

状态：GraspNet 仍用于研究和对比；主线默认 `rgbd_center`。

## 4. 右臂 IK 和轨迹姿态问题

问题：右臂末端姿态约束过强时，会出现手臂扭转、关节跳变、桌面附近斜向扫物体。

处理：

- cuRobo 负责规划到物体上方预备抓取点，最后下降尽量保持垂直。
- 使用 Kuavo 解析 IK seed 和 cuRobo 规划结果做互相审计。
- 记录每段轨迹、右夹爪末端误差、关节限制和正逆运动学诊断。
- 对最后下降加入分段路点、关节跳变阈值和水平漂移保护。

状态：能生成可执行段，但复杂物体上仍可能因机械臂末端最后下降未到达预定抓取位置而失败。最新严格物理记录 `20260626_145846` 即失败在该阶段。

## 5. 真实物理夹爪抓取不稳定

问题：严格要求物体被双指真实夹住、抬升、随末端移动时，复杂垃圾物体成功率不足。

处理：

- 调整夹爪摩擦、接触参数、闭合宽度、闭合保持时间和顶部抓取下探深度。
- 用 `gripper_physics_test.py`、`topdown_cube_grasp_test.py` 分离验证夹爪物理。
- 完整系统展示时显式启用辅助携带参数，让到达近距阈值的物体绑定到右夹爪末端后继续分类投放。
- 严格物理命令默认禁用辅助，并在失败时停止和写明原因。

状态：当前最大未闭环项。交付材料中将系统级辅助演示和严格物理夹爪验证分开统计。

## 6. 底盘和 Nav2 问题

问题：纯轮地接触模式下底盘运动不稳定，早期有 yaw 抖动和目标点残差。

处理：

- 使用 ROS2 Nav2 `NavigateToPose`，不使用手写路径作为正式导航。
- Isaac 侧采用 `wheel_drive_model=urdf_diagonal` 和 `wheel_ground_coupling=kinematic_stable`。
- 保存 `nav2_stack/` 的地图、参数、行为树和日志。
- 导航到抓取站位后重新拍照，降低残差对抓取坐标的影响。

状态：Nav2 已能完成桌边站位和桶边移动。长循环末尾仍可能出现返回观察位未能在限定时间内到达的问题，例如 `20260625_150100` 在第 9 次返回观察位失败，但此前 8 个物体处理轮次均完成投放。

## 7. 投放瞬移问题

问题：早期投放曾直接写物体 pose 到桶内，视觉上不符合“移动至桶口后释放”。

处理：

- 投放阶段先导航到目标桶侧。
- 右夹爪末端再移动到对应桶口上方。
- 打开夹爪或释放辅助携带绑定。
- 结构化记录中标明投放阶段未使用物体瞬移，并确认释放位置进入桶口水平范围。

状态：已改善。完整演示记录的已完成物体处理轮次均记录为桶口释放。

## 8. 干扰样本鲁棒性

样本：

- `trash_potted_meat_can_0`: 带食物残留语义的金属罐，可回收和厨余之间存在语义冲突。
- 历史 USD 场景中的 `trash_dirty_bottle`: 脏污瓶，测试“可回收物上沾污物”。

现象：

- `20260625_150100` 中 `trash_potted_meat_can_0` 被识别为金属罐并投到可回收桶。
- `20260626_145846` 严格单轮中同一目标分类为厨余垃圾，同时物理抓取失败。

结论：鲁棒性样本已覆盖，但分类规则还需在分类提示词和规则层明确“沾污可回收物”的判定策略，例如按比赛规则优先投其他/厨余或先清洁后可回收。
