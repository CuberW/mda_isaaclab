# Task319 视觉识别、抓取状态机与 cuRobo 讲解材料

本文档整理对象为：

`mda_isaaclab/task_319_garbage_sort/output/head_camera_grasp_records/20260625_134339`

这条记录展示了从“头部相机识别桌面垃圾”到“右臂执行物理抓取”的主流程。需要注意：这条记录的物理抓取阶段成功验证了目标被夹起，但后续运输阶段启用了 `gripper_proximity_assist` 演示辅助。因此它适合用来说明“物体位置 -> 机械臂抓取执行”的闭环流程；如果讲完整运输到垃圾桶，需要说明后续搬运不是完全依靠物理摩擦持续夹持。

## 1. 总体流程概述

这套系统可以概括为：

1. 头部 RGB-D 相机观察桌面。
2. `v28_original` 视觉链路识别垃圾物体。
3. VLM 判断物体名称、垃圾类别和目标垃圾桶。
4. YOLO11-seg 给出目标 bbox、polygon 和 mask。
5. 使用当前深度图、相机内参和 `camera_to_world` 把 2D mask 转成世界坐标下的 3D 点云。
6. 从点云中计算目标物体三维中心。
7. 根据目标位置规划机器人底座站位，使目标落入右臂可达区域。
8. 机器人导航到桌前后重新拍照，确认目标没有换。
9. 抓取状态机生成 top-down 抓取目标。
10. cuRobo 根据目标 TCP 位姿生成右臂关节轨迹。
11. 右夹爪下探、闭合、接触稳定、上抬。
12. 系统通过物体抬升量验证抓取是否成功。

这条记录中，最终选中的目标是 `trash_05`，识别名为“金属罐”，垃圾类别为“可回收物”，目标垃圾桶为蓝色可回收垃圾桶。

## 2. v28_original + VLM + YOLO11-seg 识别流程

严格来说，这里不是“YOLOv28 模型”，而是 Task319 主线中命名为 `v28_original` 的视觉链路。该链路实际使用：

- VLM：`Qwen3-VL-Flash`
- 分割模型：`YOLO11-seg`
- 模型文件：`viss/models/yolo11s-seg-best.pt`
- pipeline：`qwen_first`

本次记录中的关键配置为：

- 输入图像：`head_rgb.png`
- 图像分辨率：`1280 x 960`
- YOLO 置信度阈值：`0.15`
- ROI 扩展倍率：`2.0`
- VLM verify 模式：`all`
- Qwen 粗定位候选数：`10`
- 主检测 `detections`：`4`
- `approach_candidates`：`6`
- `rejected`：`0`

### 2.1 第一步：VLM 整图粗定位

系统先把完整的头部 RGB 图像发给 Qwen3-VL-Flash。VLM 的任务不是精确分割，而是在整张图中找出“可能是垃圾、并且可以被机器人抓取”的候选物体。

VLM 对每个候选输出：

- 候选 ID
- 物体名称
- 是否是可抓取垃圾
- 垃圾类别初判
- 归一化 2D 框
- 中心点
- 置信度
- 判断理由

prompt 中明确要求不要把机器人手臂、夹爪、桌子、地面、墙、阴影、垃圾桶边缘、传感器、支架、线缆等背景物体识别为垃圾。因此，这一步的主要价值是语义过滤和粗定位。

### 2.2 第二步：YOLO ROI 精分割

VLM 给出的 bbox 是粗框，不能直接用于抓取。系统会以 VLM bbox 为中心裁剪 ROI，并将 ROI 放大，本次放大倍率为 `2.0`。然后只在这个 ROI 内运行 YOLO11-seg。

YOLO 在 ROI 内输出更精确的：

- bbox
- polygon
- mask
- 像素中心 `centroid_px`
- 底部接触点 `bottom_contact_px`
- YOLO confidence
- final crop

如果 ROI 内出现多个 YOLO 检测，系统会综合以下指标选择最匹配 VLM 候选的实例：

- YOLO 框和 VLM 框的 IoU
- YOLO 置信度
- YOLO 检测中心与 VLM 候选中心的距离

也就是说，VLM 负责“这附近是什么”，YOLO 负责“这个物体的精确边界在哪里”。

### 2.3 第三步：VLM 单物体二次确认

YOLO 分割完成后，系统会把分割出的单物体 crop 再交给 Qwen3-VL-Flash。第二次 VLM 不再看整张图，而是只看裁剪出的主体物体。

这一步输出：

- 最终物体名称
- 垃圾类别
- 目标垃圾桶
- 是否可抓取
- 置信度
- 判断原因

在 Task319 中，YOLO 的 `raw_class_name` 通常只是 `trash_object`，它不负责垃圾细分类。真正决定“金属罐属于可回收物、应该投蓝色桶”的，是 VLM 的二次确认结果。

### 2.4 第四步：检测结果过滤和分组

系统会对候选结果做过滤、去重和分组。

正常适合规划的目标进入 `detections`。如果目标过小、太偏离图像中心，或者粗定位置信度较低，会进入 `approach_candidates`。如果不是可抓取垃圾、类别未知且不允许 unknown、或者几何信息无效，会进入 `rejected_detections`。

在本次记录中：

- 共得到 10 个 Qwen 候选。
- 4 个进入主检测 `detections`。
- 6 个进入 `approach_candidates`。
- 没有 rejected。
- 最终选中的是 `qwen_yolo_det_001`，物体名“金属罐”，类别“可回收物”。

### 2.5 第五步：2D mask 转换为 3D 物体位置

这是从视觉识别到机械臂抓取之间最关键的一步。

`v28_original` 输出的是 2D polygon、mask 和语义类别。Task319 不直接使用 VISS 自己的相机几何，而是使用当前 IsaacLab 仿真中的头部 RGB-D 相机数据：

- 当前 depth 图
- 当前相机内参
- 当前 `camera_to_world`

系统把 YOLO mask 中的有效深度像素反投影成世界坐标下的点云，然后从点云中计算物体三维中心。

本次使用的中心计算方法是：

`rgbd_point_cloud_xy_median_z_extent_midpoint`

含义是：

- XY 方向取 mask 点云的中位位置，降低噪声影响。
- Z 方向根据物体高度范围取中间值，而不是只取顶部或表面点。
- 最终得到一个更接近物体几何中心的 3D 点。

本次 reshoot 后，目标金属罐的三维中心约为：

`[1.03647, -0.15272, 0.58486] m`

有效点云数量为 `4202`，Z 向范围约 `7.76 cm`。

## 3. 抓取状态机说明

这条记录的抓取流程可以分成外层任务状态机和内层物理抓取状态机。

### 3.1 外层任务状态机

外层状态机负责从观察桌面到进入物理抓取。

1. `OBSERVE_TABLE_BACKOFF`
   机器人移动到观察位置，用头部相机观察桌面。

2. `PERCEIVE_TABLE_OBJECTS`
   执行 `v28_original` 视觉链路，识别桌面物体。

3. `PLAN_GRASP_STANDPOINT`
   根据目标三维位置规划底座站位，使目标处于右臂合适的工作空间。

4. `SELECT_NEXT_OBJECT`
   从候选物体中选择当前要抓取的目标。本次选择的是“金属罐”。

5. `NAV_TO_TABLE_STANDPOINT`
   使用导航模块将机器人移动到桌前抓取站位。

6. `FACE_TABLE_AND_SETTLE`
   面向桌子并等待仿真稳定。

7. `RESHOOT_AT_STANDPOINT`
   到达桌前后重新拍照，重新确认同一个目标的位置。

8. `PLAN_PHYSICAL_GRASP`
   根据 reshoot 后的目标三维位置生成物理抓取目标。

9. 进入内层物理抓取状态机。

重新拍照这一步很重要。机器人从观察位移动到抓取站位后，相机视角和底座位姿已经变化。如果继续使用旧坐标，抓取误差会变大。reshoot 可以在实际抓取位置重新估计目标三维中心。

### 3.2 内层物理抓取状态机

内层状态机负责真正执行右臂抓取。

1. `REST`
   右臂和夹爪进入准备状态，夹爪打开。

2. `BASE_TO_TABLE`
   确认底座已经在桌前抓取位置。

3. `STATIC_PERCEPT`
   使用当前头部 RGB-D 数据作为抓取输入。

4. `PLAN_GRASP`
   根据目标三维中心生成 top-down 抓取候选。本次候选抓取宽度约为 `5.08 cm`。

5. `PRE_GRASP`
   cuRobo 规划右臂轨迹，把右夹爪 TCP 移动到目标物体上方 hover 位。本次 pregrasp 最终 TCP 误差约 `4.2 mm`。

6. `GRASP / cartesian_linear_curobo_ik_descent`
   右夹爪沿 TCP 方向向下靠近物体。这个阶段保持 XY 对准，主要做竖直方向下探。

7. `GRASP / close_gripper`
   夹爪闭合。状态机会检查夹爪是否没有完全闭死，如果夹爪保留了一定开口，说明中间可能夹到了物体。

8. `GRASP / post_close_contact_settle`
   闭合后短暂停留，让接触稳定。

9. `LIFT`
   夹爪保持闭合，cuRobo 继续生成右臂上抬轨迹。

10. `VERIFY_HOLD`
    检查目标物体是否真的跟随夹爪上升。

11. `DONE`
    抓取状态机结束。

本次记录中，lift 验证显示目标物体被抬升约 `0.183 m`，状态机判定：

`Object lifted and held.`

因此，这一次物理抓取阶段是成功的。

## 4. cuRobo 在本环境中的用途

在 Task319 这个 IsaacLab 环境中，cuRobo 的核心作用是机械臂运动生成。它不是视觉模型，也不是垃圾分类模型，而是把目标 TCP 位姿转换成右臂可执行的连续关节轨迹。

本次抓取中，cuRobo 主要用于三个阶段：

1. 从当前右臂姿态移动到物体上方的 `pregrasp / hover` 位。
2. 从 hover 位向下靠近目标物体。
3. 闭爪后保持夹爪闭合并上抬目标物体。

它接收的主要输入包括：

- 当前右臂关节角。
- 目标 TCP 位置和姿态。
- Kuavo 右臂的 7 个关节定义。
- 关节限制。
- 规划参数和姿态约束。

它输出的是右臂 7 个关节的连续轨迹。本次右臂关节为：

`zarm_r1_joint` 到 `zarm_r7_joint`

因此，cuRobo 在这里解决的问题不是“物体在哪里”，而是“右夹爪如何从当前姿态平滑、可执行地运动到目标位置”。

## 5. cuRobo 和官方 Kuavo IK 的关系

官方 Kuavo IK 和 cuRobo 不是互相替代的关系，而是分工关系。

官方 Kuavo IK 更像是厂商提供的解析运动学模块。它可以根据目标末端位姿，计算一个符合 Kuavo 机器人模型的右臂关节解。它的优点是贴近官方机器人定义、关节命名和真实机器人的运动学约束。

cuRobo 则更像是最终运动生成器。它不仅关心末端目标位姿，也关心从当前关节状态到目标状态之间的连续轨迹。它可以检查关节限制、生成平滑轨迹，并把轨迹发送给仿真里的右臂执行。

在本系统中，官方 Kuavo IK 可以作为 cuRobo 的 seed 或校验参考。也就是说，官方 IK 先给出一个候选关节解，cuRobo 再基于这个候选解进行轨迹规划和验证。

这条记录中，官方 Kuavo IK 被调用过作为候选 seed，但对应候选解存在关节限制问题，所以没有被选中。最终实际成功执行的是 cuRobo 的右臂 pose-goal 规划。

可以这样表述：

官方 IK 负责提供符合 Kuavo 模型的候选姿态，cuRobo 负责把候选姿态或目标 TCP 转换成连续、平滑、满足限制的执行轨迹。最终发给 IsaacLab 机器人执行的是 cuRobo 生成的右臂关节轨迹。

## 6. cuRobo 在 IsaacLab 中的优点

### 6.1 当前这条 demo 中实际用到的优点

在当前 Task319 单机器人 demo 中，cuRobo 的价值主要体现在以下几点。

第一，它把“目标 TCP 位姿”转换成了“右臂连续关节轨迹”。抓取不是把手瞬移到物体上方，而是从当前姿态开始，规划出一条可以逐步执行的轨迹。

第二，它对右臂 7 个关节进行统一规划。系统不需要手写每个关节怎么转，而是给出目标 TCP，cuRobo 负责求解关节轨迹。

第三，它可以处理 pose goal、position-only 和局部笛卡尔链式下探等不同阶段。比如 pregrasp 阶段需要到达物体上方的完整位姿，下探阶段更关注 TCP 位置对准和沿指定方向靠近。

第四，它让状态机可以把动作拆成可验证的阶段。每个阶段都有规划结果、执行误差和成功标志，便于调试。例如本次 pregrasp 的最终 TCP 误差约为 `4.2 mm`，lift 阶段最终 TCP 误差约为 `5.7 mm`。

第五，它为后续迁移到真实机器人保留了更清晰的接口。视觉模块只输出目标三维位置，状态机只定义阶段目标，真正的关节轨迹由运动规划器统一生成。

### 6.2 IsaacLab + cuRobo 的通用优势

从 IsaacLab 和 cuRobo 的整体设计来看，它最大的价值在于把传统运动学规划带入 GPU 张量计算和大规模并行仿真体系中。

第一，cuRobo 具备强 GPU 并行计算能力。传统规划器通常在 CPU 上逐个求解，单次规划可能比较慢。cuRobo 可以利用 GPU 同时处理大量 IK 或轨迹优化问题，这非常适合 IsaacLab 中多环境并行仿真和强化学习训练。

第二，cuRobo 与 PyTorch 生态更容易融合。IsaacLab 中的观测、动作和奖励通常都是张量形式，cuRobo 的接口也面向 GPU 张量计算，因此更容易接入 RL 训练、环境 reset、批量目标采样和代价函数计算。

第三，cuRobo 支持高效的碰撞和距离计算。它可以用球体近似机器人几何，并在 GPU 上并行计算机器人与环境之间的距离。这对于桌面抓取、杂乱环境、多物体场景很有价值。

第四，cuRobo 可以生成更平滑的轨迹。强化学习策略直接输出动作时，动作可能抖动较大；而运动规划器生成的轨迹通常更连续，更适合后续 sim2real 迁移。

第五，cuRobo 适合做“传统规划 + 学习策略”的混合系统。比如可以用 VLM 和规则状态机决定抓哪个物体，用 cuRobo 负责到达和避障，用学习策略解决最后几厘米的接触和夹持稳定性。

需要注意的是，这些大规模并行和 RL 融合优势是 cuRobo 在 IsaacLab 中的系统级潜力。本次 20260625_134339 记录是单机器人单目标 demo，主要体现的是右臂轨迹生成、阶段化控制和抓取验证能力，并没有完全展示成百上千环境并行求解的能力。

## 7. 可直接放 PPT 的版本

本系统采用“VLM 语义识别 + YOLO 精分割 + RGB-D 三维定位 + cuRobo 右臂运动生成 + 状态机验证”的桌面垃圾抓取流程。

首先，机器人通过头部 RGB-D 相机观察桌面。RGB 图像进入 `v28_original` 视觉链路，Qwen3-VL-Flash 先在整图中进行垃圾候选粗定位，输出候选框、物体名称和垃圾类别初判。随后，系统以 VLM 候选框为中心裁剪 ROI，在 ROI 内运行 YOLO11-seg，得到更精确的 bbox、polygon 和 mask。最后，YOLO 分割出的单物体 crop 再交给 Qwen3-VL-Flash 进行二次确认，得到最终物体名称、垃圾类别和目标垃圾桶。

识别结果不会直接作为机械臂目标。系统会将 YOLO mask 投影到当前头部深度图上，结合相机内参和 `camera_to_world` 反投影成世界坐标下的目标点云，再从点云中计算物体三维中心。这样完成了从“图像中的垃圾”到“世界坐标中的可抓取目标”的转换。

在机械臂执行阶段，状态机将抓取拆分为预抓取、下探、闭爪、接触稳定、上抬和保持验证。cuRobo 根据当前右臂关节状态和目标 TCP 位姿生成连续的右臂关节轨迹，控制右夹爪从物体上方靠近、闭合并上抬。官方 Kuavo IK 可以作为候选 seed 或校验参考，用来提供符合官方机器人模型的关节解；最终执行轨迹由 cuRobo 统一规划。

本次实验中，系统识别出的目标为“金属罐”，VLM 判定为“可回收物”，对应蓝色可回收垃圾桶。机器人到达桌前后重新拍照确认目标，将目标三维中心估计为约 `[1.036, -0.153, 0.585] m`。随后右臂执行 top-down 抓取，目标物体被抬升约 `18.3 cm`，状态机判定为抓取成功。

## 8. 视频讲解词

这段视频展示的是从视觉识别到右臂物理抓取的完整流程。

首先，机器人使用头部 RGB-D 相机观察桌面。RGB 图像会进入 `v28_original` 视觉链路。这个链路不是单纯的 YOLO 检测，而是 VLM 和 YOLO 的组合：Qwen3-VL-Flash 先在整张图中找出可能是垃圾的候选物体，并判断它大概是什么垃圾；然后系统根据 VLM 给出的候选框裁剪 ROI，在 ROI 内运行 YOLO11-seg，得到更精确的物体 mask；最后再把分割出的单个物体裁剪图交给 VLM 做二次确认，确定物体名称、垃圾类别和目标垃圾桶。

在这条记录中，系统识别出了一个金属罐，并判断它属于可回收物，应该投放到蓝色可回收垃圾桶。接下来，系统不会直接用 2D 框控制机械臂，而是把 YOLO mask 结合当前深度图转换成 3D 点云，再根据点云计算物体在世界坐标系中的三维中心。

机器人随后根据这个三维位置规划桌前站位。到达站位后，系统重新拍照确认目标位置，避免使用旧视角下的坐标。确认完成后，抓取状态机开始执行。右夹爪先打开，cuRobo 根据当前右臂关节角和目标 TCP 位姿生成一条到物体上方的轨迹。到达 hover 点后，夹爪向下靠近目标，闭合夹爪，并短暂停留让接触稳定。

最后，系统让右臂保持夹爪闭合并向上抬升。它不会只根据“夹爪闭合”判断成功，而是检查目标物体是否真的跟随夹爪上升。本次记录中，目标物体被抬升了约 18.3 厘米，因此状态机判定抓取成功。

这里 cuRobo 的作用是生成右臂连续、平滑、满足关节限制的执行轨迹；官方 Kuavo IK 可以提供符合机器人模型的候选 IK 解，但最终执行仍由 cuRobo 的轨迹控制完成。

## 9. 答辩问答话术

### 问：为什么不只用 YOLO？

答：YOLO 擅长给出物体的像素级边界和 mask，但在这个任务里 YOLO 的类别通常比较泛化，比如只是 `trash_object`。垃圾分类需要语义理解，例如金属罐、牙膏、香蕉和纸盒对应不同垃圾桶，所以我们用 VLM 负责物体名称、垃圾类别和目标桶判断。

### 问：为什么不只用 VLM？

答：VLM 的语义理解能力强，但定位框通常比较粗，不适合直接控制机械臂抓取。机械臂需要精确的 mask 和几何边界，所以我们用 YOLO11-seg 在 VLM 提供的 ROI 内做精分割。

### 问：VLM 和 YOLO 谁说了算？

答：二者分工不同。VLM 决定语义类别和目标垃圾桶，YOLO 决定像素级 mask 和几何边界。简单说，VLM 负责“这是什么、投到哪里”，YOLO 负责“它的准确轮廓在哪里”。

### 问：2D 识别结果怎么变成机械臂能用的位置？

答：系统用 YOLO mask 取出深度图中对应的像素点，再通过相机内参和 `camera_to_world` 反投影到世界坐标，形成目标点云。然后从点云中计算物体三维中心，机械臂使用的是这个三维中心，而不是 2D 框中心。

### 问：为什么要到桌前后重新拍照？

答：机器人从观察位移动到桌前后，底座位姿和相机视角都发生了变化。如果继续用旧坐标，抓取误差会变大。重新拍照可以在实际抓取站位下更新目标位置，确保抓的还是同一个目标。

### 问：cuRobo 和 IK 有什么区别？

答：IK 主要解决的是一个末端位姿对应哪组关节角，是一个点解。cuRobo 解决的是从当前关节状态到目标末端位姿之间如何生成连续轨迹的问题。它不仅关心终点，还关心中间怎么过去。

### 问：为什么还要结合官方 Kuavo IK？

答：官方 Kuavo IK 更贴近 Kuavo 机器人的官方运动学定义，可以提供符合官方模型的候选关节解。我们把它作为 seed 或校验参考，再由 cuRobo 生成最终执行轨迹。这样既利用官方模型，又保留 cuRobo 的轨迹规划能力。

### 问：这次官方 IK 是否真正执行了？

答：这次官方 IK 被调用来尝试提供候选 seed，但候选解存在关节限制问题，所以没有被选中。最终成功执行的是 cuRobo 的右臂 pose-goal 规划。这也说明状态机里有容错设计，IK seed 不可用时不会强行执行。

### 问：怎么判断是真的抓住了，不只是碰了一下？

答：系统有两层判断。第一层是夹爪闭合后没有完全闭死，说明夹爪中间可能存在物体。第二层是 lift verification，也就是上抬后检查目标物体是否真的跟随夹爪升高。本次目标上升约 18.3 cm，所以判定抓取成功。

### 问：这是不是端到端学习策略？

答：不是。这一版是可解释的工程状态机：视觉给出目标位置，状态机拆分抓取阶段，cuRobo 生成机械臂轨迹，最后通过夹爪接触和物体抬升验证成功。后续如果要解决复杂接触和不同物体形状下的稳定夹持，可以在这个框架上加入学习策略。

### 问：cuRobo 在 IsaacLab 里最大的优势是什么？

答：从系统能力上看，cuRobo 最大的优势是把运动规划放到 GPU 张量计算体系里，可以更好地服务 IsaacLab 的并行仿真和强化学习训练。它可以批量求解 IK 和轨迹优化，减少 CPU 规划瓶颈，并且更容易和 PyTorch 观测、动作、奖励函数结合。

### 问：这条 demo 是否已经体现了 cuRobo 的大规模并行优势？

答：这条记录是单机器人单目标抓取 demo，主要体现的是 cuRobo 的右臂轨迹生成、阶段化执行和抓取验证能力。大规模并行 IK、批量轨迹优化和 RL 融合是 IsaacLab + cuRobo 的系统级潜力，不是这条 demo 的重点展示内容。

### 问：这条 demo 是完全物理夹持运输吗？

答：抓取和上抬阶段做了物理验证，目标确实被抬起。但这条记录后续运输阶段启用了 proximity assist 演示辅助，所以如果严格讲纯物理运输，需要把它和抓取验证阶段区分开。当前这条更适合说明“物体定位到右臂抓取执行”的状态机。

## 10. 一句话总结

这套流程的核心不是让机械臂“凭空去抓”，而是先用 VLM 和 YOLO 把桌面垃圾从图像中找出来，再用 RGB-D 把目标变成世界坐标下的三维位置，最后由状态机和 cuRobo 把这个目标位置转化为右臂可执行、可验证的物理抓取动作。
